FOR BNKR — RH NATIONAL BRIDGE (7 Tasern nation tokens onto Robinhood Chain)
===========================================================================

WHAT: The Base <-> Robinhood lane of the Tasern twin bridge, a 1:1 port of the
LIVE machinery that already runs twice (POL<->Base nation bridge, Base<->RH MfT
lane). TWO deploys: a lock vault on BASE and a mint bridge on ROBINHOOD. Users
lock a nation token's Base twin in the vault; the relayer mints an RH twin;
burning the RH twin releases the Base twin. Full flow + design rationale: PATH.md.

SUPERSEDES: tasern-bridge-rh/ (Task 2). That package's relayer note pointed at
the POLYGON vault 0xBB62...016f — but every Locked event there is ALREADY
consumed 1:1 by the live Base lane; a second consumer minting on RH from the
same events would double-mint. This lane locks on BASE instead, exactly like
the live MfT->RH lane. Coordinator promotes this package into the queue.

CREATION TX RULE (both deploys): `to` MUST be EMPTY / null / omitted —
NEVER 0x0000...0000. Verify after each: receipt.contractAddress non-null
AND eth_getCode(address) > 0.

COMPILER (both): solc 0.8.35+commit.47b9dedd, viaIR:true, optimizer 200 runs,
evmVersion paris. Source: TasernBridgeRH.sol (this folder, self-contained, no
imports). Compile script: mftusd-build/compile-rh-national-bridge.cjs.

CONSTRUCTOR ARGS: NONE for either contract (owner = deployer). Deploy tx data
= the creation bytecode file alone. See constructor-args.txt.

GAS: creation bytecodes are 2,661 bytes (vault) and 6,147 bytes (bridge) —
each deploy is well under 2M gas. Biggest post-deploy op = deployTwin (~1M,
deploys one BridgedToken). NOTHING here approaches the ~16.5M Base/RH per-tx
gas cap — no staged/router pattern needed for this package.
RH gas: maxFeePerGas 0.15 gwei / maxPriorityFeePerGas 0.01 gwei (queue rule).

POST-DEPLOY WIRING (owner-only; you are owner until the final handoff)
----------------------------------------------------------------------
STEP 1 — BASE_VAULT.setRelayer(0x849639C5D2ec97be27b90B2aC12c9d29e18f6CbC)
  Same relayer wallet as both live lanes (verified on-chain 2026-07-16 on
  0xD79360... and 0xa819b6...).

STEP 2 — BASE_VAULT.addToken(token) — one call per nation, 7 calls.
  These are the nation tokens' BASE twins (the tokens users actually hold on
  Base). Add-only allowlist.

STEP 3 — RH_BRIDGE.setRelayer(0x849639C5D2ec97be27b90B2aC12c9d29e18f6CbC)

  decimals / cap MUST mirror the Base twin — read them off the Base twin

  vault whose relayer isn't watching yet.

STEP 6 — LAST: hand ownership to the agent wallet on BOTH contracts:
  BASE_VAULT.setOwner(0xE2a4A8b9d77080c57799A94BA8eDeb2Dd6e0aC10)
  RH_BRIDGE.setOwner(0xE2a4A8b9d77080c57799A94BA8eDeb2Dd6e0aC10)
  This matches the live lane governance (verified on-chain: both live lane
  contracts are owner=0xE2a4...). Must be LAST — steps 2/4/5 are onlyOwner.

DO NOT call renounceAdminWithdraw() on BASE_VAULT. Build withdrawable,
renounce at SHIP — the Coordinator signals ship-time (queue-wide renounce
rule). It is ONE-WAY: once called, adminWithdraw is disabled forever.

DO NOT seed/lock any token value. The Coordinator runs the canary after the
relayer is up.
